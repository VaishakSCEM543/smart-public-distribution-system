# System Workflow – Smart PDS

This document describes the end-to-end dispensing workflow of the Smart PDS system — from a beneficiary arriving at the Fair Price Shop to the transaction being logged on the Authority portal.

---

## Prerequisites

- Beneficiary has a valid ration card registered in the system.
- Shop's ESP32 unit is powered on and connected to WiFi.
- Monthly quota for the current period has been allocated in Firebase.

---

## Step 1: Beneficiary Authentication

The beneficiary approaches the kiosk and enters their credentials via the **4×3 matrix keypad**.

**Supported authentication modes:**
- **OTP** — A one-time password sent to the registered mobile number or pre-generated by the authority.
- **Ration Card Number** — Manual numeric entry for fallback authentication.
- *(Planned)* **QR Code / Aadhaar Biometric** — Future phase.

**Firmware behaviour:**
1. ESP32 reads keypad input and constructs the authentication request.
2. Sends a lookup to `/customers/{id}` on Firebase RTDB.
3. If a match is found, the customer record (name, quota, consumed) is loaded into memory.
4. LCD displays: `"Auth OK"` + beneficiary name (truncated to 16 chars).

**Failure path:** If authentication fails or Firebase is unreachable, LCD shows `"! ERROR !" / "Auth Failed"` and the system returns to idle.

---

## Step 2: Entitlement Verification

After successful authentication, the ESP32 computes:

```
remaining = monthlyQuota[commodity] - consumed[commodity]
```

- If `remaining <= 0`, the LCD shows `"Quota Exhausted"` and the session ends.
- If `remaining > 0`, the system proceeds with the available balance displayed.

---

## Step 3: Commodity & Quantity Selection

The beneficiary uses the keypad to navigate a menu:

```
LCD Line 1: > Rice   Wheat
LCD Line 2:   Sugar  Kero
```

1. Press a key to select the commodity category.
2. Enter the requested quantity in kilograms (e.g., `5` for 5 kg).
3. System validates: `requested ≤ remaining`. If not, it prompts to reduce the quantity.
4. LCD confirms: `"Rice: 5.0 kg"` / `"Confirm? 1=Yes"`

---

## Step 4: Automated Dispensing (Closed-Loop)

This is the core of Smart PDS — a **sensor-verified dispensing loop** that makes over-dispensing physically impossible:

```
target_weight = requested quantity (grams)

WHILE weight_on_scale < target_weight:
    servo.open()        // allow grain to flow
    current = readWeight()
    showMessage("Dispensing...", current + "g")
    
    if (isWithinTolerance(current, target_weight)):
        servo.close()   // stop dispensing
        break

IR sensor checked → confirms actual grain flow occurred
```

- The HX711 load cell is sampled **10 times per reading** (averaged) to reduce noise.
- A 5g tolerance is applied to account for grain clumping and sensor hysteresis.
- If no weight change is detected after 30 seconds (jam detected), the servo closes and an error is logged.

---

## Step 5: Transaction Logging to Firebase

Once dispensing completes, the firmware builds a transaction record and sends it via HTTP PUT:

```json
{
  "shopId": "shop_001",
  "customerId": "cust_1042",
  "commodity": "rice",
  "requestedWeight": 5000,
  "dispensedWeight": 4987,
  "deviation": -13,
  "timestamp": 1750100500000,
  "status": "success",
  "firmwareVersion": "v2.1.3"
}
```

Simultaneously:
- `/inventory/{shopId}/rice/current` is decremented.
- `/customers/{id}/consumed/rice` is incremented.
- The transaction becomes instantly visible on the Authority portal.

---

## Step 6: Receipt & System Reset

- LCD displays: `"Done! 4987g"` / `"Thank you"`
- After 5 seconds, the system tares (re-zeros) the scale.
- ESP32 returns to the idle/authentication screen for the next beneficiary.
- Queue token is marked `"completed"` in Firebase → Shop Owner portal updates automatically.

---

## Error Handling Summary

| Condition                  | System Response                                      |
|----------------------------|------------------------------------------------------|
| WiFi lost during session   | Retries connection (up to 20 attempts); warns on LCD |
| Firebase unreachable       | Logs error to Serial; session is aborted safely     |
| Load cell reads 0 (jam)    | Servo closes; LCD shows error; logs `"failed"` tx   |
| OTP mismatch               | 3 attempts allowed, then 60-second lockout          |
| Quota already exhausted    | LCD shows message; no dispensing occurs             |
